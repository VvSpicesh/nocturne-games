## 1. 总览结论

当前程序**可以启动、可以选子走棋、可以人机对战**，达到 v0.2.3「可玩雏形」。  
对照 `CHESS_REQUIREMENTS.md` 的「稳定版 / 完整规则 / 自动存档 / 主题与计时」基线，**尚未达标**。

- 启动与基础走棋：无 **P0**
- 相对需求文档：大量 **P1**（完整规则、整盘存档、主题、计时、被吃子、真·合法走法与终局）
- AI 与模块边界：多项 **P2 / 架构问题**

## 3. 缺失功能（相对需求，按优先级）

### P1 核心缺失

1. **真·合法走法**（过滤自家王被将）及对应高亮文案承诺  
2. **王车易位**、**吃过路兵**、升变**可选**（非仅强制变后）  
3. **将军 / 将死 / 逼和**判定与状态展示、终局禁止续走  
4. **整盘自动存档 + 恢复**（§2.6 / §5，含 version）  
5. **棋盘主题**（蓝灰 + 木色）与切换按钮  
6. **被吃棋子展示**  
7. **对局计时**  
8. **AI §4**：普通一层搜索、困难 Minimax/αβ（且不卡死）  
9. AI 与人类均应基于**合法走法**候选集（当前 `allMoves` = 伪合法）

### P2 缺失 / 不完整

1. 升变无 UI 选择  
2. 标准代数记谱 / 更规范棋谱（当前为 Unicode + 箭头）  
3. 折叠屏 / 后台场景下的持久化策略（与存档绑定）  
4. storage 损坏存档降级、多版本迁移  

### P3 优化

1. `ChessEngine.applyMove` 未被 `game.makeMove` 复用，易产生双实现漂移  
2. 难度文案与真实算法不符时的产品说明（可先对齐文案，再补算法）  
3. 回归测试套件（需求 §1 要求，当前仓库无 chess 自动化回归）

## 5. 架构问题

1. **规则真相源不完整**：`engine` 应是合法局面与走子的唯一真相，但缺少 check / legalMoves / 特规状态机；`game.makeMove` 自行改盘，绕过 `applyMove`。  
2. **「合法」语义漂移**：变量名 `legalMoves`、需求文案 vs `pseudoMoves` 实现不一致，后续模块（AI、存档、测试）会建立在错误假设上。  
3. **AI 越权访问 DOM**：§3 禁止 AI 直接操作 DOM；应只返回走法，由 `game` 调 `renderer`。  
4. **storage 职责过窄**：§3 要求版本兼容与整盘读写；现状像 preference helper，不是对局存档层。  
5. **状态字段不足**：无 `castlingRights`、`enPassant`、`captured`、`theme`、`clock`、`gameOver` 等，无法干净演进到需求基线。  
6. **拆分与行为**：framework 拆分本身保留了 v0.2.3 行为；缺口主要来自**相对需求文档尚未实现**，不是拆分导致的文件丢失。对照 `legacy/chess/` 与当前模块化代码，基础走棋能力等价。

## 6. 建议修复顺序

在**不重写项目、不搬目录**前提下，按投入产出与依赖关系：

| 序 | 项 | 级别 | 理由 |
|----|----|------|------|
| 1 | `engine`：检测将军 + `legalMoves`（过滤伪走法）；`game`/`AI` 改用合法集；高亮名实相符 | P1 / B1 | 规则底盘，后面特规/终局/AI 都依赖 |
| 2 | `engine`：将死 / 逼和 + `game` 终局停手 + `renderer` 提示 | P1 / B2 | 对局可「结束」 |
| 3 | `engine`：易位、过路兵；升变可选（可最小：对话框三键） | P1 | 完整规则体 |
| 4 | 统一走子入口：`makeMove` → `engine.applyMove`（含特规） | P3→P1 预防 | 避免双实现 |
| 5 | `storage`：versioned 整盘存档；`game` 在走棋/悔棋/翻转/`visibilitychange`/`pagehide` 触发；`init` 恢复 | P1 / B6 | 稳定版可恢复 |
| 6 | `renderer` + CSS + `index.html`：被吃子列表、主题切换（蓝灰/木色） | P1 | 需求 2.4 / 2.5 |
| 7 | 对局计时（状态 + 存档字段 + UI） | P1 | 可后于存档字段设计一并加 |
| 8 | 修正记谱时机（升变前符号） | P2 / B3 | 小改 |
| 9 | `ai`：剥离 DOM；简单/普通/困难按 §4（困难限深 + 超时保险） | P1 / B4 B5 | 依赖合法走法与 `applyMove` 模拟 |
| 10 | 手工/脚本回归用例（开局、被将不得走、易位、刷新恢复） | P3→支撑 §1 | 锁基线 |

## 7. 恢复到稳定版 — 最小修改文件清单

此处「稳定版」指：**继续沿用现有目录与模块边界**，把行为抬到需求文档可验收的最小闭环（能合法对局、能终局、能刷新恢复；主题/计时/完整 AI 可分批）。

### 7.A 达到「规则正确 + 可恢复」的最小必改文件

必须改（核心闭环）：

1. `chess/js/engine.js` — 合法走法、将军/将死/和、易位、过路兵、统一 `applyMove`  
2. `chess/js/game.js` — 使用合法走法、终局、触发存档、串事件  
3. `chess/js/storage.js` — versioned 整盘存读 + 损坏降级  
4. `chess/js/renderer.js` — 将军/终局文案、（可选同期）吃子展示对接  
5. `chess/js/ai.js` — 只消费合法走法；去掉直接 DOM；难度算法按档升级  

强烈建议同期改（否则 2.4/2.5 仍不达标）：

6. `chess/index.html` — 主题按钮、吃子区、计时 DOM  
7. `chess/css/board.css` — 主题色变量 / 木色格  
8. `chess/css/themes.css` — 主题相关棋子若需微调  
9. `chess/css/ui.css` — 吃子条、计时、主题按钮样式  

一般**不必**为恢复稳定版而改：

- `shared/*`（大厅共用）  
- `mahjong/*`  
- `legacy/*`（只作对照，不作为运行入口）  
- 为「恢复」而再搬迁目录或合并回单文件 — **不建议**（违背当前框架；且非必要）

### 7.B 若目标仅为「回到拆分前同等可玩度」（非需求全文）

当前 `chess/js/*` + `chess/css/*` 已与 `legacy/chess/` 行为基本等价（可玩雏形）。  
**无需为恢复可玩而回滚文件**；需求缺口应用 **7.A** 增量修补，而不是整体还原 monolithic `chess.js`。

对照备份路径（只读参考）：

- `legacy/chess/engine.js`  
- `legacy/chess/chess.js`  
- `legacy/chess/chess.css`  
- `legacy/chess/index.html`

### 7.C 不要做的事（本审计约束重申）

- 不要为审计结果立刻重写项目  
- 不要移动现有模块目录  
- 不要在本阶段「开新架构」；在现有五文件 + 三 CSS 内补齐
